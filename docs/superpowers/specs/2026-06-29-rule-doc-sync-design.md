# Rule Doc Sync — Design

Date: 2026-06-29
Status: approved (design, post-review), pending implementation

## Goal

Generate a natural-language conventions markdown file from the configured
`RULES` list, so coding agents can read the rules they must follow instead of
reverse-engineering them from `enforcer_config.py`. Keep the file in sync with
the config automatically — block commits that let it drift.

## Motivation

Today the only human/agent-facing description of the rules is
`enforcer/docs.py:render_rules_markdown` (`enforcer/cli.py:279` `docs`
command). Its output is a mechanical field dump:

```markdown
## branch-naming

**Severity:** ERROR
**Message:** Branch '{matched_value}' doesn't match required pattern: type/description
**File globs:** *
**Fix:** Rename: git branch -m <type>/<description>
```

This is a spec sheet, not a set of instructions an agent can follow. Three
problems follow:

1. **No rationale.** An agent told "no `print()`" will write
   `sys.stdout.write("...")` — satisfies the matcher, defeats the intent.
   Rules without their *why* invite workarounds.
2. **No auto-sync.** `enforcer docs -o CONVENTIONS.md` is manual and
   undocumented in the repo flow. The README mentions it once (`README.md:81`)
   but no `CONVENTIONS.md` is tracked. Rules drift from any prose doc.
3. **No self-enforcement.** Nothing blocks a commit that updates
   `enforcer_config.py` and forgets to regenerate the doc.

This design closes all three gaps with one deterministic renderer, one CLI
command, one new `Rule` field, and one self-enforcement rule.

## Non-goals

- **LLM-composed prose.** Non-deterministic, costs tokens, harder to
  diff-review. The doc must stay in sync with structured config; deterministic
  templates guarantee that by construction. Existing `LLMMatcher`/`llm.py`
  infra is not used here.
- **Replacing the hand-written `AGENTS.md`.** Its architecture map, matcher
  contract, and workflow notes are not derivable from `RULES`. Stays
  hand-written; gets one pointer line to the generated doc.
- **Merging generated content into `AGENTS.md`.** Considered and rejected —
  in-place block replacement in a tracked file is trickier and risks
  clobbering hand edits. A separate file is cheaper and the drift surface is
  smaller.
- **A `prose` override field on `Rule`.** Originally proposed alongside
  `rationale`; dropped as YAGNI. Stripped `message` covers the "what",
  `rationale` covers the "why" — a third "what, but nicer" field is
  nice-to-have, not needed.
- **A `--check` flag on `sync-doc`.** The `DocSyncMatcher` handles enforcement
  in both pre-commit and CI. A separate CLI `--check` duplicates the matcher's
  logic with no current caller. Dropped; add later if a standalone CI step is
  needed.
- **A `rule-needs-rationale` self-enforcement rule.** After the initial
  backfill (rollout step 1), it never fires on existing rules. New rules
  without rationale are a code-review concern, not a runtime check. The matcher
  would re-parse config the runner already loaded, and as a WARN it's
  self-suppressable via `--confirm-read-warnings`. Dropped as YAGNI.
- **Import-time doc writes or file-watcher daemons.** Both are over-engineered
  for a problem a one-line CLI call solves.

## Design

### 1. New `Rule` field: `rationale`

File: `enforcer/rule.py`

```python
@dataclass
class Rule:
    ...
    rationale: str = ""   # why this rule exists; prevents agent workarounds
```

Defaults to `""` — backward-compatible, no existing rule breaks. Rules with
empty `rationale` simply omit the `**Why:**` block in the generated prose.

This is the substance of the feature. The renderer is mechanical; the
rationales are what stop agents from gaming the matchers. Example:

- `no-print`: "`print()` writes to stdout, which is reserved for
  machine-readable output in CLI tools. Mixing human prose into stdout breaks
  piping and scripting. `sys.stderr` is the correct channel for
  human-facing diagnostics."
- `function-max-params`: ">5 params signals the function does too much; group
  into a dataclass to make the boundary explicit."
- `no-secrets`: "Hardcoded secrets ship to the repo and can't be rotated
  without a commit; env vars separate config from code."

All 25 existing rules in `enforcer_config.py` get backfilled with one
rationale sentence each.

### 2. Prose renderer: `render_rules_doc` in `enforcer/docs.py`

Second function in the existing `enforcer/docs.py` (which already holds
`render_rules_markdown`). No new module — both functions are `list[Rule] -> str`
with different templates; the split would be unjustified.

```python
def render_rules_doc(rules: list[Rule], *, workspace: str | None = None) -> str:
    """Render configured rules as natural-language markdown instructions."""
```

**Output structure:**

```markdown
# Conventions

_26 rules configured. Auto-generated by `enforcer sync-doc`. Do not edit by hand._

## ERROR — must fix before commit

### branch-naming

Branch names must match `^(feature|fix|chore|docs|refactor)/`. If they don't,
rename: `git branch -m <type>/<description>`.

**Why:** Branches encode intent; CI/greps depend on the `type/` prefix to
route checks and changelogs.

**Applies to:** enforcer_config.py, CONVENTIONS.md

**Fix:** Rename: git branch -m <type>/<description>

---

### class-capwords
...

## WARN — blocks unless acknowledged

### verify-types-changed
...

## INFO — advisory

...
```

**Rendering logic:**

1. Group rules by severity in fixed order: ERROR, WARN, INFO.
2. Within each group, sort by `id` (deterministic, matches `render_rules_markdown`).
3. Each rule renders as:
   - `### {id}` heading.
   - One imperative sentence: the `message` field with `{placeholders}`
     stripped. Guard for callable messages first:
     `msg = "(dynamic message)" if callable(rule.message) else rule.message`,
     then `re.sub(r"\{[a-zA-Z_][a-zA-Z0-9_]*\}", "", msg)` and collapse
     double spaces. The regex matches Python-identifier-style placeholders
     (`{matched_value}`, `{file}`, `{line}`, `{column}`) and is robust
     against future placeholders with digits or uppercase.
   - `**Why:**` block — the `rationale` field verbatim, if
     `rationale.strip()` is non-empty.
   - `**Applies to:**` line — file globs verbatim (e.g. `enforcer/**/*.py`).
     No human-form translation — keeps the renderer trivial and the output
     unambiguous. METADATA rules get the suffix "(metadata rule, checked once
     per run)".
   - `**Excludes:**` line if `exclude_globs` non-empty.
   - `**Fix:**` line — `fix_instruction` verbatim, if non-empty.
   - `**Read targets:**` line if `read_targets` non-empty.
   - `**LLM check:**` line if `llm_consequence` present (prompt + model).
   - `diff_only` rules get a note: "Checked on changed lines only (`--staged`)."
   - `---` separator between rules within a group.

**Determinism guarantees:**

- No timestamps, no environment-dependent output.
- Rule count appears once in the header (derived from `len(rules)`) — changes
  only when the rule set changes, which is exactly when the doc should change.
- Same `RULES` → byte-identical output across two calls. Verified by a test.
- Sorting is stable; no dict-ordering dependence.

**Placeholder stripping detail:** the `message` field is written for runtime
match reporting, not prose. It contains `{matched_value}`, `{file}`, `{line}`,
`{column}`. The renderer strips these with
`re.sub(r"\{[a-zA-Z_][a-zA-Z0-9_]*\}", "", msg)` and collapses double spaces.
This is a known simplification — some messages read slightly awkwardly without
the placeholder (e.g. "Function '' at must be snake_case."). The fix is to
tighten the message in `enforcer_config.py` during the rationale backfill
(rollout step 1), not to add a `prose` field.

### 3. CLI command: `enforcer sync-doc`

File: `enforcer/cli.py`

```python
@cli.command(name="sync-doc")
@click.option("--config", "config_path", default="enforcer_config.py")
@click.option("--output", "-o", default="CONVENTIONS.md")
def sync_doc(config_path, output):
    """Regenerate the natural-language conventions doc from configured rules."""
    from enforcer.docs import render_rules_doc
    from enforcer.config import load_config

    config = load_config(config_path)
    fresh = render_rules_doc(config.rules, workspace=config.workspace)

    _assert_output_contained(output, config.workspace or ".")
    Path(output).write_text(fresh, encoding="utf-8")
    click.echo(f"Wrote {output}")
```

- Default `--output CONVENTIONS.md` (configurable — user explicitly asked for
  this to not collide with `AGENTS.md`).
- No `--check` flag. Enforcement is handled by the `DocSyncMatcher` (section 5),
  which fires in both pre-commit (`--staged`) and CI (`--all`) mode.
- Reuses `_assert_output_contained` (`cli.py:286`) — keeps the existing
  path-escape guard consistent across all write commands.
- `docs` command (`cli.py:279`) stays unchanged — it's the mechanical
  renderer, still useful for raw field inspection. The two commands coexist.

### 4. `AGENTS.md` pointer line

One line added near the top of `AGENTS.md`, after the title:

```
> Auto-generated rule list: see CONVENTIONS.md (`enforcer sync-doc` to regenerate).
```

No other changes to `AGENTS.md`. The hand-written sections (architecture map,
matcher contract, branch/commit conventions) stay — they describe things not
derivable from `RULES`.

### 5. Self-enforcement rule: `conventions-md-stale`

One new rule in `enforcer_config.py`. Catches drift in both directions:
config changed without regenerating the doc, or the doc edited directly.

```python
Rule(
    id="conventions-md-stale",
    severity=Severity.ERROR,
    matchers=[DocSyncMatcher(config_path="enforcer_config.py", doc_path="CONVENTIONS.md")],
    file_globs=["enforcer_config.py", "CONVENTIONS.md"],
    message="CONVENTIONS.md is stale or missing. Regenerate after changing rules.",
    fix_instruction="Run: enforcer sync-doc",
    rationale="A stale conventions doc misleads agents — they follow rules that "
              "no longer match the actual config. The doc must be regenerated "
              "whenever RULES changes, and direct edits to CONVENTIONS.md must "
              "not drift it from the config.",
),
```

Matcher: `enforcer/matchers/doc_sync.py`

```python
@dataclass
class DocSyncMatcher:
    """Flags if the on-disk generated doc differs from a fresh render.

    Reads config rules from shared_ctx["__rules__"] (set by the CLI runner),
    falling back to load_config(self.config_path) when called standalone.
    Reads CONVENTIONS.md from self.doc_path on disk.
    """
    config_path: str
    doc_path: str
    needs: Needs = Needs.RAW

    def find(self, file_ctx: FileContext, shared_ctx: dict | None = None) -> list[Match]:
        shared_ctx = shared_ctx or {}
        # Prefer rules already loaded by the runner; fall back for standalone use.
        rules = shared_ctx.get("__rules__")
        workspace = shared_ctx.get("__workspace__", ".")
        if rules is None:
            from enforcer.config import load_config
            config = load_config(self.config_path)
            rules = config.rules
            workspace = config.workspace or "."
        from enforcer.docs import render_rules_doc
        fresh = render_rules_doc(rules, workspace=workspace)
        # Read on-disk doc; missing file = stale (emits a match).
        try:
            on_disk = Path(self.doc_path).read_text(encoding="utf-8") if Path(self.doc_path).exists() else ""
        except OSError:
            on_disk = ""
        if on_disk != fresh:
            return [Match(file=file_ctx.path, line=0, rule_id="conventions-md-stale",
                          message="CONVENTIONS.md is stale or missing.", matched_value=self.doc_path)]
        return []
```

**Key design decisions:**

1. **Bidirectional drift detection.** `file_globs=["enforcer_config.py",
   "CONVENTIONS.md"]` — fires when either file is staged. The comparison is
   the same in both cases: render fresh doc from config, compare against
   on-disk CONVENTIONS.md. Catches config-changed-without-regen AND
   direct-edits-to-CONVENTIONS.md.

2. **No bare `except` around `load_config`.** Per AGENTS.md matcher contract
   rule #4 ("no try/except around core logic"), `load_config` errors propagate.
   A broken `enforcer_config.py` should fail loudly, not silently pass the
   sync check. The only `try/except` is around the `CONVENTIONS.md` file read
   (`OSError`) — that's an external file IO failure, consistent with the
   contract's "only catch around external calls."

3. **Reuses loaded config via `shared_ctx["__rules__"]`.** The CLI runner
   already loaded the config (`cli.py:243`). Stashing `config.rules` in
   `shared_ctx["__rules__"]` (one-line addition to `_build_shared_ctx` at
   `cli.py:156`) lets the matcher skip a redundant `importlib` round-trip.
   Falls back to `load_config(self.config_path)` when called standalone
   (e.g. in tests).

4. **On-disk, not staged.** The matcher reads `enforcer_config.py` and
   `CONVENTIONS.md` from the working tree, not the staged version. In the
   normal workflow (stage → commit immediately), on-disk == staged. The
   narrow window (stage, then edit without re-staging) is documented here as
   a known limitation — the matcher validates the working tree state, not the
   index. Fixing this would require reading `git show :enforcer_config.py`
   which is a larger change for a narrow edge case.

5. **`diff_only=False` (the default).** The rule fires on any change to the
   watched files, not just changed lines. In `--staged` mode, `file_globs`
   filtering ensures it only runs when `enforcer_config.py` or
   `CONVENTIONS.md` is staged. In `--all` mode (CI), it runs against both
   files. Per `rule.py:58-60`, `diff_only=False` rules with
   `changed_lines is None` still fire — confirmed correct for CI.

Edge case: first run, `CONVENTIONS.md` doesn't exist. `on_disk = ""`, which
won't equal `fresh` → emits a match telling the dev to run `enforcer sync-doc`
once. Stale-or-missing is the same violation; no special case.

### 6. `shared_ctx["__rules__"]` plumbing

File: `enforcer/cli.py`, function `_build_shared_ctx` (line 156).

One-line addition:

```python
def _build_shared_ctx(config, builder, ws: str) -> dict:
    shared_ctx: dict = {}
    shared_ctx["__rules__"] = config.rules       # <-- new
    shared_ctx["__workspace__"] = config.workspace or ws   # <-- new
    for rule in config.rules:
        ...
```

This makes the already-loaded rules available to any matcher that needs them,
not just `DocSyncMatcher`. `__workspace__` carries the resolved workspace
root for matchers that need a consistent path reference.

### 7. `__init__.py` exports

- `enforcer/matchers/__init__.py`: add `DocSyncMatcher` to `__all__` and
  imports.
- `enforcer/rule.py`: `rationale` field is a plain dataclass field, no export

## Testing

Per AGENTS.md "Testing Minimum Bar" — every new matcher ships with paired
tests. Every new core function ships with test coverage.

| New code | Test file |
|---|---|
| `render_rules_doc` in `enforcer/docs.py` | `tests/test_docs.py` (add cases) |
| `enforcer/matchers/doc_sync.py` | `tests/test_matchers/test_doc_sync.py` |
| `Rule.rationale` field | covered by `tests/test_rule.py` (add case) |
| `enforcer sync-doc` CLI | covered by `tests/test_cli.py` (add cases) |
| `shared_ctx["__rules__"]` | covered by `tests/test_cli.py` (add assertion) |

**`tests/test_docs.py` cases (render_rules_doc):**
- Empty rules → header + "No rules configured."
- Single rule with `rationale` → `### id`, `**Why:**`, `**Applies to:**`, `**Fix:**`
- Rule without `rationale` (empty string) → no `**Why:**` line
- Rule with whitespace-only `rationale` (`"  "`) → no `**Why:**` line (`.strip()` guard)
- Rule with callable `message` → `"(dynamic message)"` in prose, no `TypeError`
- Rule with `llm_consequence` → `**LLM check:**` line
- `diff_only=True` → "Checked on changed lines only" note present
- Rules grouped by severity (ERROR before WARN before INFO)
- Within group, sorted by `id`
- Determinism: two calls with same input → byte-identical output
- Placeholder stripping: `Function '{matched_value}' at {file}:{line}` → "Function '' at must be snake_case." (placeholders removed, double spaces collapsed)

**`tests/test_matchers/test_doc_sync.py` cases:**
- In-sync file → no matches
- Stale file (content differs) → one match
- Missing file → one match (treated as stale)
- `shared_ctx["__rules__"]` present → uses it (no `load_config` call)
- `shared_ctx["__rules__"]` absent → falls back to `load_config(self.config_path)`
- `load_config` raises → error propagates (not swallowed)

**`tests/test_cli.py` cases (sync-doc):**
- `sync-doc --output FILE` writes the rendered doc
- `sync-doc` default output is `CONVENTIONS.md`
- `_build_shared_ctx` stashes `__rules__` and `__workspace__`
